# 记录作者的心路历程...

## 第一天
用 maas-glm-5.1-zhipl Assistant 模式，开启了项目的立项，在经过了简单的一些问答后，Agent就开始帮我干活了，但我注意到他一规划完就马上想开始给我生成代码，遂制止了他。 开始引导他生成PRD文档。

![alt text](PixPin_2026-06-27_16-37-08.png)
不过这倒也不能怪他，毕竟我没有给够足够的上下文和约束条件。脑袋里其实也还没有清晰的想法。本来想和他多进行几轮对话的，结果一看PRD好家伙感觉我看了一下，只对AI相关的内容感兴趣，遂又拿着PRD和他进行讨论。 但我发现这样的约束，我要是一个字一个字的敲给他，岂不是很累人，所以还不如去网上找优秀的规则文档直接丢给他，让他学完之后再帮我生成质量好的代码。

然后这时候我又想到，正是因为一开始没有想清楚需求痛点和功能，他不知道我想要什么，作为一个更倾向执行的AI工具来说，他能看到的就是快速满足用户的表面需求：快速生成一个东西交给用户。 所以造成了我们对PRD进行返工的情况。 

所以还是要先规划好我的需求到底想做什么，再设计好原型，最后到代码实现层才不会造成过多的返工。 毕竟好的模型算力token现在是很贵的...其实就是减少成本。


## 6.29 开始用codex
不同模型生成出来的不太一样，GPT5.5明显比glm-5.1推理能力更强，我只是把同样的叙事求给GPT，他很明显比glm更能get到我的意思，并且生成的PRD我也较为满意，能想到我想不到的，又能get到我的主要意思，而glm为了生成不出错还是经常会反问我一些问题。 不过也有可能codex提前看到了glm生成的PRD，所以更能理解项目的本意。

接下来想找一些优秀的设计稿，好好把原型打磨一下，或者在网上找一些优秀的实践(插件 or skill)，不然生成半天效果不好又要返工。对了还要把git分支好好管理下，以及哪些是GPT做的哪些是glm做的，后续可能还会考虑接入deepseek的能力。


## 7.1 第一次崩溃
果然不出意外的出意外了，jobpilot由于受到boss直聘风控的影响，导致连一个最小的poc功能都跑不通，就是一键给hr发送预先填充好的信息。其他的一键总结分类jd这个我就先不谈了，关键的投递简历我也先搁一边去，但是至少打招呼怎么着也得先做出来吧。不然这个插件还有啥意思，还好我机智提前设想了可行性的问题，要是真当我把全部原型、高保真、设计文档准备好再开始实施的话，那就有点为时已晚了。(不过好像确实也做的差不多了)

好了 经过本人的调试，发送消息的功能终于已经做好了，就是很别扭地需要先填入再二次确认发送，并且这么简单的功能居然要我调试这么久，并且我发现，人boss直聘上已经内置了一个常用语的对话框，这我怎么没想到！=-= 所以要不还是回到最初步的想法，那就是AI一键帮我检索我需要投递的岗位，而不用我手动筛选然后再一个个点开查看，然后再帮我同时给所有公司的hr打招呼。

好了 经历了和codex多轮对话 我的第一个梦想idea破碎了， 以下是codex给我的回复

```
现在真正的瓶颈
没有 BOSS 官方 API
如果没有官方投递/打招呼接口，我们就没法直接、安全、稳定地说：“给这 20 个 HR 发招呼。”
所谓自动化只能落到网页交互层，也就是你说的“模拟人”。

网页自动化天然脆
BOSS 是一个复杂 SPA 页面。聊天框、发送按钮、岗位详情、弹窗、已沟通过状态、风控提示，都不是稳定接口。页面一变，选择器就失效。你看到的“关掉聊天窗口”“打开文件夹”，本质就是插件在黑盒页面里误判了控件。

真正的一键批量发送会碰风控
从平台视角看，“短时间打开多个岗位并发送类似招呼语”非常像自动化行为。即使技术上做出来，也可能遇到验证码、频控、账号限制。

Chrome 插件不是后台托管执行器
插件可以辅助当前页面，但要跨页面、跨状态、连续执行任务，必须做完整状态机、错误恢复、页面识别、风控暂停、日志回放。现在这个 PoC 还只是粗糙插件，不是成熟执行器。

发送消息是外部副作用
这不是本地按钮，而是真的给 HR 发消息。要做到“用户知道你在做什么，关键时刻 yes/no”，需要一个托管任务系统，而不是在页面上临时塞几个按钮。

所以结论是：你想要的版本现在还没做到。
更准确地说：
“半自动填入/发送”：已经初步可行。
“批量自动打招呼”：技术上可以继续试，但当前实现不可靠。
“低干预托管式自动投递”：目前没有验证通过。
“稳定商业化”：在没有官方 API 的情况下，风险很高。
如果继续做，下一步不该继续堆按钮了。应该重构成一个真正的 JobPilot 托管执行器 PoC：
用户先选一批岗位。
JobPilot 展示计划：准备联系哪些岗位、用什么话术。
用户一次性确认。
执行器逐个处理。
每一步识别页面状态。
遇到异常暂停，让用户 yes/no 决策。
全程有日志和失败原因。
默认低频，不追求海投。
但我也要说清楚：只要没有 BOSS API，它底层仍然是网页自动化。区别只是从“瞎点按钮”升级成“有状态、有诊断、有暂停机制的网页执行器”。
如果你问“为什么这么简单还做不到”，答案是：
因为对我们来说它看起来只是发一句话，但对技术实现来说，这是在一个不受控第三方平台上，稳定执行带真实外部副作用的自动化任务。没有 API，就只能和页面结构、风控、登录态、弹窗、验证码打架。
当前 PoC 的结论我会修正为：
Conditional Go 偏弱：单次辅助发送可行，但批量托管未验证通过。继续投入前，必须先做执行器级 PoC，而不是继续修这个小插件。
```
