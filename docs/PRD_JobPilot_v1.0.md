## 一、文档概述

### 1.1 文档目的

本文档定义 JobPilot 的产品目标、用户场景、功能边界和 MVP 范围。JobPilot 的核心不是“AI 简历诊断”，而是帮助求职者在招聘平台上更高效地筛选岗位、制定投递策略、批量触达 HR，并处理高频沟通。

### 1.2 产品定位

**一句话描述**：JobPilot 是一款 AI 求职投递效率工具，帮助求职者自动理解 JD、分类岗位、制定投递节奏，并在招聘平台上辅助完成打招呼、跟进和面试沟通。

**核心判断**：

- 求职低效的主要原因不是简历不会写，而是岗位信息阅读成本高、投递节奏混乱、重复沟通耗时。
- 产品经理、运营、市场等岗位的 JD 差异大，同名岗位在不同公司可能要求完全不同，需要 AI 帮用户快速分类和筛选。
- 很多求职者会先投递不那么心仪的小公司、远地区或低优先级岗位练手，再投递目标公司，JobPilot 应该服务这种真实投递策略。
- 用户在上班、通勤或忙碌时，仍希望后台持续推进投递和沟通，而不是人工逐个阅读、打招呼、回复。

### 1.3 非目标

JobPilot v1 不以“简历美化/简历诊断”为核心卖点。

- 不做复杂简历编辑器。
- 不把 AI 简历评分作为首页主功能。
- 不承诺替用户凭空生成不真实经历。
- 不鼓励无差别骚扰式投递，而是基于用户规则和平台限制做受控触达。

### 1.4 术语定义

| 术语 | 说明 |
|------|------|
| JD | Job Description，职位描述和任职要求 |
| 岗位分类 | AI 根据 JD 将岗位归入方向、难度、匹配度、优先级等标签 |
| 投递策略 | 用户定义或 AI 建议的投递顺序，例如先练手、再冲刺目标岗位 |
| 打招呼 | 在 Boss 直聘等平台向招聘者发起沟通 |
| 自动投递任务 | 根据筛选条件和节奏规则，批量执行打招呼或投递动作 |
| 话术库 | 面向 HR 高频问题的可复用回答模板 |

## 二、用户与场景

### 2.1 目标用户

| 用户类型 | 特征 | 核心痛点 |
|----------|------|----------|
| 在职跳槽者 | 白天仍在工作，可用于求职的时间碎片化 | 没时间逐个阅读 JD、打招呼、回复 HR |
| 产品/运营/市场类岗位求职者 | 岗位方向细分多，JD 差异大 | 同样叫“产品经理”，实际要求可能是 AI 产品、B 端、增长、数据、商业化等不同方向 |
| 有策略投递意识的求职者 | 会先练手，再投目标公司 | 投递顺序、岗位优先级和沟通状态容易混乱 |
| 海投型求职者 | 需要快速扩大沟通量 | 重复筛选和重复问答消耗大量时间 |

### 2.2 典型用户故事

1. 作为一个在职 AI 产品经理候选人，我希望 AI 能批量阅读 Boss 上的 JD，把岗位分成 AI 应用产品、平台产品、数据产品、商业化产品等方向，避免我逐个点开阅读。
2. 作为一个想先找手感的求职者，我希望系统能先投递低优先级公司、非目标城市或小公司，等沟通和面试节奏稳定后，再提醒我投递目标公司。
3. 作为一个白天上班的人，我希望手机后台可以按规则帮我打招呼，并把 HR 回复集中整理出来，让我只处理真正需要决策的消息。
4. 作为一个频繁被 HR 问相同问题的人，我希望系统能根据预设资料自动生成或推荐回答，例如到岗时间、期望薪资、项目经验、面试时间。

## 三、核心价值主张

1. **少读 JD**：AI 批量提取岗位要求、方向、关键词和风险点，让用户快速判断是否值得聊。
2. **会排优先级**：根据公司、城市、薪资、岗位方向、匹配度和用户偏好生成投递队列。
3. **少重复操作**：把逐个打招呼、标记状态、记录 HR 回复等重复动作自动化。
4. **少重复沟通**：对高频 HR 问题提供固定话术、变量填充和自动回复建议。
5. **省心托管**：用户完成关键设置后，AI 默认自动筛选、打招呼和整理回复，只在异常、高风险或强主观决策时打断用户。

## 四、产品形态

| 端 | 定位 | 核心场景 |
|----|------|----------|
| 移动端/小程序 | **主体验入口** | 完成关键设置、开启 AI 求职托管、查看今日进展、处理少量异常和 HR 回复 |
| Web 端 | **辅助查看入口** | 查看岗位分析、投递效果、回复率、约面率和策略复盘 |
| 浏览器插件/自动化助手 | **后台能力入口** | 在招聘平台上下文中读取 JD、同步岗位、执行打招呼和消息同步 |

MVP 阶段优先围绕 Boss 直聘场景设计；其他平台作为后续扩展。

## 六、MVP 功能清单（Phase 1）

### 6.1 F1 - 用户求职偏好配置

| 项目 | 说明 |
|------|------|
| 功能描述 | 用户配置目标岗位、城市、薪资、公司偏好和投递禁区 |
| 优先级 | P0 |
| 端 | Web + 移动端 |

**需求细节**：

- 支持配置目标岗位方向，例如 AI 产品经理、B 端产品经理、增长产品经理、数据产品经理。
- 支持配置目标城市、可接受城市、暂不考虑城市。
- 支持配置目标薪资、最低接受薪资、是否接受外包/创业公司/小公司。
- 支持配置公司黑名单、行业黑名单、关键词黑名单。
- 支持配置投递目标：练手优先、均衡投递、目标公司优先。

**验收标准**：

- [ ] 用户可以创建并保存一套求职偏好。
- [ ] 岗位筛选和投递策略能读取偏好配置。
- [ ] 黑名单岗位不会进入自动投递队列。

### 6.2 F2 - AI JD 分类与摘要

| 项目 | 说明 |
|------|------|
| 功能描述 | AI 批量阅读 JD，输出岗位方向、要求摘要、匹配度和标签 |
| 优先级 | P0 |
| 端 | Web + 移动端 |

**需求细节**：

- 输入：岗位标题、公司、城市、薪资、完整 JD。
- 输出岗位方向：AI 产品、B 端产品、C 端产品、增长产品、数据产品、项目型产品、运营偏产品、其他。
- 输出核心要求：经验年限、行业背景、能力关键词、工具/技术要求、加分项。
- 输出风险标签：外包疑似、销售属性强、岗位职责模糊、强加班暗示、薪资不透明。
- 输出匹配度：高/中/低，并解释主要原因。
- 输出阅读摘要：用 3-5 条短句说明这个岗位到底要什么人。

**验收标准**：

- [ ] 批量导入 JD 后可以生成分类结果。
- [ ] 每个岗位至少包含方向、摘要、关键词、风险标签、匹配度。
- [ ] 用户可以手动修正 AI 分类，修正结果被保存。

### 6.3 F3 - AI 求职托管

| 项目 | 说明 |
|------|------|
| 功能描述 | 根据用户偏好和 AI 分类自动执行筛选、打招呼和节奏控制 |
| 优先级 | P0 |
| 端 | 移动端为主，Web 辅助查看 |

**策略模板**：

| 模板 | 说明 |
|------|------|
| 练手优先 | 先投递非目标城市、小公司、低优先级岗位，用于找沟通和面试手感 |
| 稳妥推进 | 高中低优先级混合投递，保持稳定回复量 |
| 目标冲刺 | 优先投递高匹配、高薪资、目标公司岗位 |
| 自定义 | 用户配置筛选条件、排序规则和每日上限 |

**需求细节**：

- 用户完成目标岗位、城市、薪资、公司偏好、避雷关键词和投递节奏等关键设置后，可一键开启 AI 托管。
- AI 自动按匹配度、公司优先级、城市、薪资、发布时间、风险标签生成执行顺序，无需用户逐条确认。
- 支持每日打招呼上限、每小时上限、随机间隔和工作时间段限制。
- 自动跳过已投递、已沟通、黑名单、薪资低于下限、外包疑似和高风险岗位。
- 仅在验证码/风控、薪资超范围、高风险岗位、HR 问题不确定时进入“需要我看”。
- 用户可随时暂停托管、调整规则或手动接管。

**验收标准**：

- [ ] 用户完成关键设置后可以开启 AI 托管。
- [ ] AI 可自动筛选岗位并执行打招呼，不需要逐条确认。
- [ ] 不符合规则的岗位会自动跳过并记录原因。
- [ ] 异常和不确定事项会进入“需要我看”，不阻塞其他自动执行。
- [ ] 用户可以随时暂停、继续或调整托管规则。

### 6.4 F4 - 一键打招呼/辅助投递

| 项目 | 说明 |
|------|------|
| 功能描述 | 在用户授权后，按照投递队列辅助完成打招呼动作 |
| 优先级 | P0 |
| 端 | 浏览器插件/自动化助手 + 移动端进度查看 |

**需求细节**：

- 支持为不同岗位方向生成不同打招呼文案。
- 文案包含岗位相关性，不使用完全重复模板。
- 支持用户设置默认招呼语和禁用词。
- 支持记录执行结果：成功、失败、自动跳过、需要用户处理。
- 支持遇到验证码、登录失效、平台风控提示时自动暂停并通知用户。
- 所有自动动作必须在用户授权的账号和范围内执行。

**验收标准**：

- [ ] 用户开启托管后，系统可以按规则自动执行打招呼。
- [ ] 每条执行结果被记录。
- [ ] 出现异常时任务自动暂停，不继续盲目执行。
- [ ] 用户可以查看已打招呼岗位列表。

### 6.5 F5 - HR 消息聚合与回复建议

| 项目 | 说明 |
|------|------|
| 功能描述 | 聚合 HR 回复，识别问题类型并生成可直接发送的话术建议 |
| 优先级 | P0 |
| 端 | 移动端 + Web |

**高频问题类型**：

- 什么时候方便面试
- 期望薪资是多少
- 当前是否离职、到岗时间
- 是否有某类项目经验
- 是否接受某城市/通勤/出差
- 简单介绍一下过往经历

**需求细节**：

- 用户预先维护个人回答资料，包括期望薪资、到岗时间、可面试时间段、核心项目经历。
- AI 根据 HR 消息识别问题类型。
- 对确定性问题直接生成回复建议。
- 对不确定问题标记“需要我看”，由用户确认后再发送。
- 支持一键复制、发送前编辑。
- MVP 阶段允许对确定性高频问题生成可直接发送的回复建议；是否自动发送由用户在话术设置中授权。

**验收标准**：

- [ ] HR 消息能按岗位和公司聚合展示。
- [ ] 系统能识别至少 5 类高频问题。
- [ ] 回复建议能引用用户预设资料。
- [ ] 用户可以编辑后发送或复制。

### 6.6 F6 - 投递看板与效果复盘

| 项目 | 说明 |
|------|------|
| 功能描述 | 跟踪投递、沟通、约面状态，并分析策略效果 |
| 优先级 | P1 |
| 端 | Web + 移动端 |

**状态定义**：

| 状态 | 含义 |
|------|------|
| 待筛选 | 已采集但未完成 AI 分类 |
| 待投递 | 已进入投递队列，尚未打招呼 |
| 已打招呼 | 已向 HR 发起沟通 |
| 已回复 | HR 已回复 |
| 约面中 | 正在协调或已确认面试 |
| 已结束 | 不合适、无回复超时、用户放弃 |

**验收标准**：

- [ ] 岗位状态随任务执行自动更新。
- [ ] 用户可以手动修改状态。
- [ ] 能展示回复率、约面率、不同策略的效果对比。

## 八、合规与安全边界

- 用户必须明确授权平台账号、投递范围和执行频率。
- 系统必须提供暂停、取消、黑名单、执行日志和“需要我看”入口。
- 不绕过平台安全机制；遇到验证码、风控、登录异常时暂停。
- 不生成虚假经历、不替用户承诺不真实信息。
- 敏感信息加密存储，包括简历、手机号、薪资期望、聊天记录。
- 对自动回复功能默认分级：确定性固定问题可授权自动回复，不确定或涉及承诺的信息必须进入“需要我看”。

## 十、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 招聘平台限制自动化行为 | 核心动作无法稳定执行 | 降低频率、异常暂停、插件辅助，不绕过风控 |
| AI 分类不准 | 投递队列质量下降 | 支持人工修正、保存反馈、迭代 Prompt |
| 过度自动化导致误投 | 用户体验和信任受损 | 关键设置前置，黑名单、薪资下限、风险标签和异常确认兜底 |
| 自动回复不合适 | 影响求职沟通 | MVP 默认只生成建议，固定问题才允许用户开启自动回复 |
| 隐私数据泄露 | 法律和信任风险 | 加密存储、最小化采集、访问审计 |
